# Повторные попытки запроса
[English](../en/retry.md) | [简体中文](../cn/retry.md) | [繁體中文](../tw/retry.md) | [日本語](../ja/retry.md) | [한국어](../ko/retry.md) | [Español](../es/retry.md) | [Türkçe](../tr/retry.md) | **Русский**

Компонент поддерживает глобальные повторные попытки и декларативные повторные попытки.

## Глобальные повторные попытки

Глобальные повторные попытки по умолчанию отключены, конфигурация по умолчанию:

```yaml
retrofit:
  global-retry:
    # Включить ли глобальные повторные попытки
    enable: false
    # Базовый интервал глобальных повторных попыток (миллисекунды)
    interval-ms: 100
    # Максимальное количество глобальных повторных попыток
    max-retries: 2
    # Стратегия退避: FIXED (фиксированный интервал, по умолчанию) / EXPONENTIAL (экспоненциальный退避)
    backoff-strategy: fixed
    # Верхний предел интервала экспоненциального退避 (миллисекунды), действует только для EXPONENTIAL
    max-interval-ms: 30000
    # Коэффициент джиттера [0.0, 1.0], 0.0 означает отсутствие джиттера
    jitter: 0.0
    # Правила глобальных повторных попыток
    retry-rules:
      - response_status_not_2xx
      - occur_io_exception
```

### Правила повторных попыток

Правила повторных попыток поддерживают три конфигурации:

1. **RESPONSE_STATUS_NOT_2XX**: выполнять повторную попытку, когда код состояния ответа не 2xx
2. **OCCUR_IO_EXCEPTION**: выполнять повторную попытку при возникновении IO-исключения
3. **OCCUR_EXCEPTION**: выполнять повторную попытку при возникновении любого исключения

### Стратегия退避 и джиттер

`backoffStrategy` управляет способом увеличения интервала повторных попыток, по умолчанию `FIXED` совпадает с историческим поведением:

- **FIXED**: интервал каждой повторной попытки фиксирован и равен `intervalMs`
- **EXPONENTIAL**: экспоненциальный退避, интервал N-й повторной попытки = `intervalMs × 2^N` (N начинается с 0), ограничен `maxIntervalMs`, чтобы избежать бесконечного увеличения интервала

`jitter` (значение `[0.0, 1.0]`, по умолчанию `0.0` — без джиттера) используется для добавления случайного джиттера к вычисленной задержке, чтобы избежать эффекта thundering herd при синхронных повторных попытках от нескольких клиентов:

> Фактическая задержка = вычисленная задержка × (1 + jitter × random), где random — случайное число в диапазоне `[0, 1)`

### Условный триггер: по коду состояния / типу исключения

На основе грубых правил `RetryRule` можно дополнительно сузить условия триггера (по умолчанию пусто, совпадает с историческим поведением):

- `retryStatusCodes`: выполнять повторную попытку только при совпадении кода состояния ответа с списком (необходимо использовать вместе с правилом `RESPONSE_STATUS_NOT_2XX`). Например `{502, 503, 504}`
- `retryExceptionClasses`: выполнять повторную попытку только при совпадении типа исключения с списком (дополнительно сужает на основе исключений, соответствующих `RetryRule`). Например `{SocketTimeoutException.class}`

```java
@RetrofitClient(baseUrl = "http://localhost:8080/")
@Retry(maxRetries = 3, intervalMs = 200, backoffStrategy = BackoffStrategy.EXPONENTIAL,
        maxIntervalMs = 5000, jitter = 0.3, retryStatusCodes = {502, 503, 504})
public interface Api {
    @GET("getUser")
    User getUser(@Query("id") Long id);
}
```

## Декларативные повторные попытки

Если повторные попытки нужны только для части запросов, можно использовать аннотацию `@Retry` на соответствующих интерфейсах или методах.

## Пользовательское расширение

Если нужно изменить поведение повторных попыток запроса, можно наследовать `RetryInterceptor` и зарегистрировать его как Spring Bean.
