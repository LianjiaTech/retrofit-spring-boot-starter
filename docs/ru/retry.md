# Повторные попытки запросов
[English](../en/retry.md) | [简体中文](../cn/retry.md) | [繁體中文](../tw/retry.md) | [日本語](../ja/retry.md) | [한국어](../ko/retry.md) | [Español](../es/retry.md) | [Türkçe](../tr/retry.md) | **Русский**

Компонент поддерживает глобальные повторные попытки и декларативные повторные попытки.

## Глобальные повторные попытки

Глобальные повторные попытки по умолчанию отключены, конфигурация по умолчанию:

```yaml
retrofit:
  global-retry:
    # Включить ли глобальные повторные попытки
    enable: false
    # Базовый интервал повторных попыток (мс)
    interval-ms: 100
    # Максимальное количество повторных попыток
    max-retries: 2
    # Стратегия экспоненциального роста: FIXED (фиксированный интервал, по умолчанию) / EXPONENTIAL (экспоненциальный рост)
    backoff-strategy: fixed
    # Верхний предел интервала экспоненциального роста (мс), действует только при EXPONENTIAL
    max-interval-ms: 30000
    # Коэффициент jitter [0.0, 1.0], 0.0 означает отсутствие jitter
    jitter: 0.0
    # Глобальные правила повторных попыток
    retry-rules:
      - response_status_not_2xx
      - occur_io_exception
```

### Правила повторных попыток

Поддерживаются три конфигурации правил:

1. **RESPONSE_STATUS_NOT_2XX**: Повторная попытка выполняется, если код состояния ответа не 2xx
2. **OCCUR_IO_EXCEPTION**: Повторная попытка выполняется при возникновении IO-исключения
3. **OCCUR_EXCEPTION**: Повторная попытка выполняется при возникновении любого исключения

### Стратегия экспоненциального роста и jitter

`backoffStrategy` управляет способом увеличения интервала повторных попыток, по умолчанию `FIXED` (соответствует историческому поведению):

- **FIXED**: Интервал каждой повторной попытки фиксирован и равен `intervalMs`
- **EXPONENTIAL**: Экспоненциальный рост, интервал N-й повторной попытки = `intervalMs x 2^N` (N начинается с 0), ограничен `maxIntervalMs`, чтобы предотвратить бесконечное увеличение интервала

`jitter` (значение `[0.0, 1.0]`, по умолчанию `0.0` -- без jitter) добавляет случайное колебание к расчетной задержке, чтобы избежать эффекта thundering herd при синхронных повторных попытках с нескольких клиентов:

> Фактическая задержка = расчетная задержка x (1 + jitter x random), где random -- случайное число в диапазоне `[0, 1)`

### Условный запуск: по коду состояния / типу исключения

На основе грубых правил `RetryRule` можно дополнительно сузить условия запуска (по умолчанию пусто, соответствует историческому поведению):

- `retryStatusCodes`: повторная попытка выполняется только при совпадении кода состояния ответа с списком (необходимо использовать с правилом `RESPONSE_STATUS_NOT_2XX`). Например: `{502, 503, 504}`
- `retryExceptionClasses`: повторная попытка выполняется только при совпадении типа исключения с списком (дополнительно сужает исключения, соответствующие `RetryRule`). Например: `{SocketTimeoutException.class}`

```java
@RetrofitClient(baseUrl = "http://localhost:8080/")
@Retry(maxRetries = 3, intervalMs = 200, backoffStrategy = BackoffStrategy.EXPONENTIAL,
        maxIntervalMs = 5000, jitter = 0.3, retryStatusCodes = {502, 503, 504})
public interface Api {
    @GET("getUser")
    User getUser(@Query("id") Long id);
}
```

## Декларативные повторные попытки

Если повторные попытки нужны только для некоторых запросов, используйте аннотацию `@Retry` на соответствующем интерфейсе или методе.

## Кастомное расширение

Если нужно изменить поведение повторных попыток, можно наследовать `RetryInterceptor` и зарегистрировать его как Spring Bean.
